¿Qué es REST?

Representational State Transfer - El estilo arquitectónico que domina las APIs web modernas

Definición Simple

REST significa Representational State Transfer (Transferencia de Estado Representacional).

Es un estilo arquitectónico para diseñar APIs web que sigue principios específicos para hacerlas escalables, mantenibles y fáciles de usar.

¡No es un protocolo!

REST es un conjunto de principios y restricciones, no un protocolo como HTTP. Aunque casi siempre se usa con HTTP.

GET

Obtener datos

POST

Crear datos

PUT

Actualizar datos

DELETE

Eliminar datos

Lo esencial: REST organiza todo como recursos (usuarios, productos, pedidos) que se manipulan con verbos HTTP.

La Analogía del Restaurante RESTful

1
Menú (Recursos): Cada plato es un recurso (/hamburguesas, /pizzas, /bebidas)
2
Pedir (GET): "Quiero ver el menú de hamburguesas" = GET /hamburguesas
3
Ordenar (POST): "Quiero una hamburguesa con queso" = POST /pedidos
4
Modificar (PUT): "Cambio la hamburguesa por una con bacon" = PUT /pedidos/123
5
Cancelar (DELETE): "Cancelar mi pedido" = DELETE /pedidos/123

Stateless: Cada pedido es independiente. El mozo no recuerda tus pedidos anteriores (sin estado).

Los 6 Principios de REST

Para que una API sea considerada RESTful, debe seguir estos principios:

Arquitectura Cliente-Servidor

Separación clara entre frontend y backend

Stateless (Sin estado)

Cada solicitud contiene toda la información necesaria

Cacheable

Las respuestas deben indicar si se pueden cachear

Sistema en Capas

El cliente no sabe si hay servidores intermedios

Código bajo demanda

(Opcional) El servidor puede enviar código ejecutable

Interfaz uniforme

El principio más importante. Tiene 4 sub-principios:

Interfaz Uniforme - Las 4 Reglas

1. Identificación de recursos: Todo es un recurso con URL única (/usuarios/123)

2. Manipulación a través de representaciones: Trabajas con JSON/XML, no con el recurso directo

3. Mensajes autodescriptivos: Cada mensaje tiene toda la info necesaria

4. Hipermedia como motor de estado: Links a otros recursos en las respuestas

REST vs SOAP

Las dos principales formas de construir APIs web:

REST

Estilo arquitectónico

  • Usa HTTP directamente
  • JSON/XML liviano
  • Sin estado
  • Más rápido y simple
  • Ideal para web/móvil

SOAP

Protocolo

  • Protocolo XML estricto
  • WSDL para contrato
  • Con estado
  • Más robusto y seguro
  • Ideal para empresarial

¿Cuándo usar cada uno?

REST

APIs públicas, móviles, startups

SOAP

Bancos, sistemas legacy, alta seguridad

¡Importante hoy!

REST es el estándar actual: El 90% de las APIs nuevas son REST. SOAP se usa principalmente en sistemas empresariales existentes.

Códigos de Estado HTTP en REST

Cada respuesta REST incluye un código que indica qué pasó:

Éxito (2xx)

200 OK
201 Created
204 No Content

Significado: La solicitud se completó exitosamente

Redirección (3xx)

301 Moved
304 Not Modified

Significado: Se necesita acción adicional

Error Cliente (4xx)

400 Bad Request
401 Unauthorized
404 Not Found

Significado: Error del lado del cliente

Error Servidor (5xx)

500 Internal Error
503 Service Unavailable

Significado: Error del lado del servidor

GET /api/productos/999
404 Not Found
{
  "error": "Producto no encontrado",
  "message": "El producto con ID 999 no existe",
  "statusCode": 404
}

Versión de API

/api/v2/usuarios

Siempre versiona tu API para mantener compatibilidad cuando hagas cambios grandes.

REST en Microservicios

En arquitectura de microservicios, cada servicio expone su propia API REST. Esto permite que los servicios se comuniquen entre sí de manera estandarizada:

  • Servicio Usuarios: REST API en /api/usuarios
  • Servicio Pedidos: REST API en /api/pedidos
  • Servicio Pagos: REST API en /api/pagos

¡Cada microservicio es independiente pero se comunican mediante APIs REST bien definidas!

POST /api/v1/pedidos
Content-Type: application/json
{
  "usuarioId": 123,
  "productos": [456, 789],
  "total": 29990
}
201 Created
{
  "id": 1001,
  "status": "pendiente",
  "_links": {
    "self": "/api/v1/pedidos/1001",
    "usuario": "/api/v1/usuarios/123"
  }
}
